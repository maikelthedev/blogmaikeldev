Section 5: Page Pre-Rendering and Data Fetching

This is about SSR (server-side rendering) which is the key feature of NextJs.

NextJS have to types of pre-rendering, static generation and SSR on the fly.

Using getStaticProps()

export async function getStaticProps(context) {}

The function must be written exactly like that and it’ll only execute on the server, never on the client, in fact it’ll be hidden from the client. There’s no window object. You can put DB code there for example. Or read from a filesystem file (useful for markdown or JSON for example). Use the FS module with async/await.

https://nodejs.org/api/fs.html#promise-example

It should return {props: whateverObject}

You can tell that function to work again on

You can say it with the props that you return, example

return {
        props: someObject, 
        revalidate: 10 // This would be th enumber of seconds to revalidate. 
        }

That’s incremental static regeneration.

Using redirect.

It’s another key inside getStaticProps where you coudl specify where to send the user in case the page didn’t load or the access to the DB didn’t work for any reason.

return {
    redirect: '/404-nodata
    }

Extracting context for SSR.

Instead of route.query.eventId (just recycling the example above, you’d use inside the getStaticProps this:

export async function getStaticProps(context) {
    const { params } = context;
    const eventId = params.eventId;
}

Using getStaticPath()

Default behavior for dynamic pages is not to generate them, such as [eventId]

export async function getStaticPaths() {
    return {
        paths: [
                {params: {eventId: "1"} }
                {params: {eventId: "2"} }
                {params: {eventId: "3"} }
        ],
        fallback: false
    }
}

Using fallback: true

Using fallback set to true stuff is generated before when first visited. Imagine one as before but with eventId:4. When using this you should put a loading page or something because nothing will be created no the first visit yet.

Not found

You can return {notFound: true} to send the user to a 404 Not found page in the absence of a product with the ID the user request for example.

Using getServerSideProps()

This one is never statically pre-generated. You get access to params, req and res. Yoo cannot use it with getStaticProps because it wouldn’t make any sense.

export async function getServerSideProps(context) {
const {params, req, res} = context
returns { props: thePropsObject} 
}

What’s useSWR hook.

Is one to use instead of useEffect() to get data from the client side. Just check the docs.

Section 8: Adding Backend Code with API Routes (Fullstack React)

The code inside “api” won’t ever go to the client.

You must use the API folder

NextJS requires that if you use an API it has the folder named “api”. It’s very similar to vanilla express but also uses the pages file-based format. So if you want a users api endpoint you just create a users.js file inside “api” folder and inside, similar as express:

function handler(req, res) {
    res.status(200
        .json({status: "ok"})
}

Communication front-end backend

If you’re going to use getStaticProps from the frontend then you shouldn’t use fetch since it’s all in your project. Instead you should DRY and make the code you have in your API a function that can be accessed from getStaticProps()
